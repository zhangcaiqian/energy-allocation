# 留白（Blanc）— 产品方案文档

> 版本：v1.0 | 日期：2026-02-11 | 作者：zhangjun

---

## 1. 产品概述

### 1.1 一句话描述

为精力透支的互联网从业者提供一个以主观感受为核心的精力觉察工具，通过"精力配置"框架帮助用户像管理资产一样管理精力，在忙碌中留出呼吸的空间。

### 1.2 产品背景

- **核心问题**：互联网从业者长期处于高强度脑力劳动中，缺乏对自身精力状态的觉察，经常在透支后才意识到问题。市面上的效率工具都是"以任务为本"（关注你还有多少事没做完），缺乏"以人为本"（关注你现在感觉怎么样）的精力管理产品。
- **目标用户**：25-35 岁互联网从业者（程序员、产品经理、设计师、运营），MVP 阶段先聚焦程序员群体。
- **为什么是现在**：职业倦怠问题日益严重（WHO 已将 Burnout 列入 ICD-11），中国互联网行业高强度工作文化下，精力透支已成为普遍问题，但缺乏有效的自我觉察工具。

### 1.3 产品愿景

成为互联网人的"精力觉察伙伴"——不催你做更多，而是帮你意识到什么时候该停下来。用温暖而非焦虑的方式，帮助用户建立可持续的精力管理习惯。

### 1.4 核心理念

借鉴资产配置的思路：**永远保留 30%-40% 的精力用于恢复**，就像投资中永远不满仓一样。这个比例可由用户自行调整（20%-50%）。

### 1.5 Slogan

> 你今天留白了吗？

## 2. 市场分析

### 2.1 目标用户画像

| 维度 | 描述 |
|------|------|
| 人群特征 | 25-35 岁互联网从业者，高强度脑力劳动，经常加班，会议密集 |
| 核心痛点 | 长期精力透支、下班后"被掏空"、周末也恢复不过来、缺乏对自身精力的系统觉察 |
| 使用场景 | 午休时、下班后记录精力状态；查看精力趋势发现规律；精力低谷时获得关怀 |
| 现有应对 | 靠咖啡硬撑、周末报复性睡觉、偶尔冥想，但缺乏系统性管理 |
| 设备习惯 | 手机重度用户，Mac/iPhone 为主，对新工具接受度高 |
| 付费意愿 | 对好用的工具有付费习惯（Notion、各种效率工具订阅者） |

**用户细分差异**：
- **程序员**：精力消耗模式是"长时间深度专注 → 突然断崖式疲惫"
- **产品/运营**：模式是"高频切换上下文 → 持续低水平消耗"
- **管理层**：模式是"大量会议 → 情绪劳动导致的隐性消耗"

### 2.2 竞品分析

| 竞品 | 定位 | 优势 | 劣势 | 与留白的差异 |
|------|------|------|------|-------------|
| Bearable | 多维度健康追踪 | 维度丰富 | 太复杂，像填病历 | 留白只聚焦精力，极简 |
| Rise Science | 基于睡眠的精力预测 | 科学依据强 | 过度依赖睡眠单一维度 | 留白以主观感受为核心 |
| 番茄钟/Forest | 时间管理 | 用户基数大 | 以任务为本，不关心人的状态 | 留白以人为本 |
| 潮汐/Calm | 正念冥想 | 品牌好 | 只解决"恢复"不解决"觉察" | 留白覆盖觉察+配置+恢复 |
| Gyroscope | 健康数据仪表盘 | 数据全面 | 价格贵、信息过载 | 留白轻量、温暖、聚焦 |

### 2.3 差异化定位

市面上没有一款产品同时满足以下三点：
1. 以**主观体感**（而非客观硬件数据）为核心数据源
2. 提供**精力配置**框架（保留比例概念）
3. 以**温暖陪伴**而非效率驱动的方式呈现

**竞争优势**：留白不需要替代任何现有工具——它可以和番茄钟（管时间）、Calm（管放松）共存，在用户的工具箱里占据一个全新的位置。

## 3. 产品方案

### 3.1 MVP 范围定义

**核心假设**：

| # | 假设 | 验证标准 |
|---|------|---------|
| 1 | 用户愿意每天花几秒钟记录精力状态 | 7 日留存率 > 30% |
| 2 | "精力配置"框架能帮用户做出更好的精力分配决策 | 用户自评"精力管理有改善" > 50% |
| 3 | 温暖陪伴型的 AI 教练能提升用户持续使用的意愿 | 周均 check-in ≥ 3 次 |

### 3.2 核心用户故事

1. 作为互联网打工人，我想要每天花几秒记录精力状态，以便觉察自己的精力消耗模式
2. 作为容易透支的人，我想看到自己的精力"花园"状态，直观知道该继续投入还是休息
3. 作为想改善精力管理的人，我想看到一段时间的精力趋势，发现规律（如"每周三最累"）
4. 作为长期疲惫的人，我想在精力低谷时收到温暖关怀和恢复建议，而不是被催着做更多

### 3.3 功能列表与优先级

| 功能 | 优先级 | 描述 | 验证假设 |
|------|--------|------|---------|
| 精力 Check-in | Must | 温暖问题 + 4 级状态选择（🟢🟡🟠🔴），5 秒完成 | 假设 1 |
| 3D 花园仪表盘 | Must | 首页 Three.js 花园场景，精力值驱动植物状态变化（塞尔达风格化写实） | 假设 2 |
| 精力配置设定 | Must | 设定精力保留比例（默认 30%），支持 20%-50% 调整 | 假设 2 |
| LLM 精力教练 | Must | check-in 后温暖回应、低谷关怀、模式洞察 | 假设 3 |
| 自定义 check-in 时间 | Must | 用户可在设置页自定义 check-in 时间点 | 假设 1 |
| 周/月趋势图 | Should | 折线图展示精力变化，标记工作日/周末 | 假设 2 |
| 每周精力报告 | Should | LLM 生成周度精力回顾和建议 | 假设 3 |
| 社交/排行榜 | Won't | MVP 不做 | - |
| 可穿戴设备联动 | Won't | MVP 不做 | - |
| 团队精力看板 | Won't | MVP 不做 | - |

### 3.4 关键用户流程

**首次使用**：
```
欢迎页 → 3 页引导（介绍"留白"精力配置理念）→ 设定保留比例
→ 邮箱注册 → 完成首次 check-in → 看到花园 + AI 回应 → 进入首页
```

**日常核心循环**：
```
打开留白 → 看到花园当前状态 → 点击 check-in
→ 看到一个温暖的问题 → 选择精力状态（5 秒）
→ 花园动画变化 + AI 教练回应 → 关闭 / 查看趋势
```

**精力 Check-in 设计**：
- 不采用精确数字打分（1-10），而是用**温暖的问题 + 4 级选项**
- 问题从预设模板池（80+ 条）中按时间段随机选取，避免重复
- 4 级选项：🟢 精力充沛 / 🟡 状态尚可 / 🟠 有点疲惫 / 🔴 快没电了

### 3.5 产品信息架构

```
/                → 首页（3D 花园 + check-in 入口 + 教练回应）
/check-in        → check-in 流程页（问题 + 选择）
/trends          → 精力趋势图（周/月切换）
/settings        → 设置（保留比例、check-in 时间、账户）
```

## 4. 技术方案

### 4.1 技术选型

| 层级 | 技术方案 | 选择理由 |
|------|---------|---------|
| 框架 | Next.js 14+ (App Router) | 前后端一体，部署简单，全栈同一项目 |
| 样式 | TailwindCSS + shadcn/ui | 快速出好看的 UI，组件质量高 |
| 3D 场景 | React Three Fiber + @react-three/drei | Three.js 的 React 声明式封装，开发效率高 |
| 3D 模型 | Poly.pizza / Sketchfab 低多边形模型 | CC-BY 协议，用 gltf.pmnd.rs 转 React 组件 |
| 3D 风格 | 塞尔达风格化写实（MeshToonMaterial + Bloom 后处理） | 温暖有氛围感 |
| 数据库 | Turso (SQLite) | 零运维，免费额度充足（9B 行读取/月） |
| ORM | Drizzle ORM | 轻量、类型安全，支持 SQLite/Turso |
| 认证 | Auth.js + 邮箱密码 | 简单通用，MVP 阶段不做忘记密码和邮箱验证 |
| LLM | 通义千问（Qwen） | 成本低，Function Calling 稳定，OpenAI 兼容接口 |
| 部署 | Vercel | Next.js 原生支持，免费 Hobby Plan |
| PWA | next-pwa | 支持安装到手机主屏幕 |

### 4.3 核心数据模型

```typescript
// 用户表
User {
  id                  String    // UUID
  email               String    // 登录邮箱
  passwordHash        String    // bcrypt 加密
  name                String
  energyReserveRatio  Float     // 精力保留比例，默认 0.3
  checkInTimes        String    // 自定义 check-in 时间，如 "09:00,12:00,18:00,22:00"
  createdAt           DateTime
}

// 精力记录表（核心表）
EnergyCheckIn {
  id            String    // UUID
  userId        String    // → User.id
  level         Enum      // HIGH / MEDIUM / LOW / EXHAUSTED
  question      String    // 当次 check-in 的问题
  note          String?   // 用户可选备注
  aiResponse    String?   // LLM 即时回应
  checkInAt     DateTime  // 记录时间
}

// 每日精力摘要表（预计算）
DailyEnergySummary {
  id            String
  userId        String    // → User.id
  date          String    // "YYYY-MM-DD"
  avgScore      Float     // 日均精力值
  minScore      Float     // 日最低
  maxScore      Float     // 日最高
  checkInCount  Int       // 记录次数
  belowReserve  Int       // 低于保留线的次数
  createdAt     DateTime
}

// LLM 教练消息表
CoachMessage {
  id            String
  userId        String    // → User.id
  triggerType   Enum      // CHECK_IN_REPLY / LOW_ENERGY_ALERT / WEEKLY_SUMMARY
  content       String    // LLM 生成的内容
  createdAt     DateTime
}
```

### 4.4 精力值映射

| 等级 | 标签 | 数值 | 视觉 |
|------|------|------|------|
| HIGH | 精力充沛 | 1.0 | 🟢 |
| MEDIUM | 状态尚可 | 0.65 | 🟡 |
| LOW | 有点疲惫 | 0.35 | 🟠 |
| EXHAUSTED | 快没电了 | 0.1 | 🔴 |

### 4.5 LLM 集成架构

**混合方案**：

| 场景 | 策略 | 原因 |
|------|------|------|
| check-in 即时回应 | 预组装 Context（今日记录 + 近 3 天摘要）→ 单次调用 | 数据确定，追求速度 |
| 精力教练深度场景 | Function Calling（LLM 自主取数据）→ 多轮调用 | 需求不确定，追求灵活 |
| 每周精力报告 | Function Calling | 需要多步推理和异常深挖 |

**Function Calling Tools 定义**：

| Tool | 功能 | 参数 |
|------|------|------|
| `get_today_checkins` | 获取今天所有精力记录 | 无 |
| `get_daily_summary` | 获取某天精力摘要 | date: string |
| `get_energy_trend` | 获取近 N 天趋势 | days: number |
| `get_weekly_summary` | 获取周度精力数据 | week: "current" / "previous" |
| `get_user_settings` | 获取用户精力配置 | 无 |

**LLM 调用约束**：
- 最大工具调用轮次：3 轮
- check-in 回复最大 token：150
- 周报最大 token：500
- 超时降级：LLM 调用失败时使用预设兜底话术

### 4.6 3D 花园场景

**精力值驱动场景变化**：

| 精力状态 | 花园表现 | 技术实现 |
|---------|---------|---------|
| 🟢 充沛 | 花朵盛开、色彩鲜艳、轻轻摇曳、阳光明媚 | scale=1.0, 明亮 ToonMaterial, Float 动画, 暖色 Environment |
| 🟡 尚可 | 花朵半开、颜色柔和、微风 | scale=0.7, 柔和材质, 慢速动画 |
| 🟠 疲惫 | 花朵低垂、颜色偏暗、安静 | scale=0.5, 暗淡材质, 几乎静止 |
| 🔴 耗竭 | 花朵闭合、灰色调、需要浇水 | scale=0.3, 灰色材质, 静止, 夜晚 Environment |

**性能优化**：
- `frameloop="demand"`：不需要每帧渲染，状态变化时才重渲染
- `React.lazy` 懒加载 3D 场景，避免阻塞首屏
- 低多边形模型，单个 GLB 控制在 500KB 以内，场景总计不超过 2MB

### 4.7 技术风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| 3D 场景开发超时 | 高 | Plan B：先用静态插画/Lottie 动画，3D 后补 |
| 移动端 3D 性能问题 | 中 | frameloop=demand + 低多边形 + 懒加载 |
| LLM 回应质量不稳定 | 中 | 兜底话术模板 + try-catch 降级为规则回复 |
| Function Calling 格式出错 | 低 | 降级为预组装 Context 模式 |
| 千问 API 不可用 | 低 | 备用 DeepSeek，接口兼容可快速切换 |

## 5. 可行性评估

### 5.1 评估总览

| 维度 | 评级 | 说明 |
|------|------|------|
| 技术可行性 | **高** | 全栈开发者 + AI 辅助编码，技术栈成熟，无技术壁垒 |
| 市场可行性 | **中高** | 痛点真实、有空白地带，但精力管理类产品留存率普遍偏低，需验证 |
| 商业可行性 | **中** | MVP 阶段不考虑变现，先验证留存；长期订阅制可行 |
| 团队可行性 | **高** | 独立开发者最适合小而美工具产品，决策快、迭代快 |

### 5.2 风险矩阵

| 风险 | 发生概率 | 影响程度 | 应对策略 |
|------|---------|---------|---------|
| 用户记录几天就放弃（留存差） | 高 | 致命 | LLM 教练持续提供价值 + 花园养成感 + 温暖文案降低负担 |
| 3D 花园场景开发超时 | 中 | 高 | 准备 Plan B 降级为 2D 方案 |
| "精力"概念太抽象，用户不知道怎么选 | 中 | 高 | 用具象化问题引导，而非直接打分 |
| LLM 回应不够温暖/跑偏 | 中 | 中 | 精心调试 System Prompt + 兜底话术 |
| 产品沦为"另一个日记工具"无差异化 | 中 | 高 | 强化精力配置框架的独特价值，让"保留线"成为核心体验 |

### 5.3 关键决策点

| 时间 | 决策 | 依据 |
|------|------|------|
| Day 3 | 3D 场景是否降级为 2D | 如果进度严重落后 |
| 上线后第 7 天 | 是否继续投入 | 7 日留存率是否 > 30% |
| 上线后第 30 天 | 是否开始做付费功能 | 周活跃率是否稳定 |

## 6. 运营与增长策略

### 6.1 冷启动计划

- **种子用户来源**：身边的互联网从业者朋友
- **第一批用户获取方式**：微信群直接邀请，发链接 + "你今天留白了吗？"
- **早期运营方式**：
  - 建微信群，每天定时提醒 check-in（人工运营，不做推送）
  - 在群里收集反馈，快速迭代
  - 目标：先拉 20-30 人稳定使用

### 6.2 获客渠道（MVP 验证后）

| 渠道 | 策略 | 优先级 |
|------|------|--------|
| 即刻 / Twitter | 分享产品故事 + 精力管理理念 | 高 |
| V2EX / 掘金 | 独立开发者经验分享 | 高 |
| 小红书 | 职场健康 / 精力管理话题 | 中 |
| 口碑传播 | 产品内生成可分享的精力周报卡片 | 中 |

### 6.3 留存策略

- **核心留存指标**：周活跃 check-in 率（每周 ≥ 3 次记录的用户占比）
- **关键留存动作**：
  - **花园养成感**：持续记录 → 花园状态越来越好，形成正反馈
  - **AI 教练个性化洞察**：用得越久越懂你，让用户感到"被理解"
  - **周度精力报告**：每周一份有价值的回顾，形成打开习惯
  - **留白理念内化**：让"保留精力"成为用户的思维习惯

### 6.4 数据指标体系

- **北极星指标**：周活跃 check-in 率（每周 ≥ 3 次记录的用户占比）
- **关键过程指标**：
  - 激活：注册后完成首次 check-in 率 > 80%
  - 留存：7 日留存率 > 30%
  - 深度：周均 check-in ≥ 3 次
  - 满意度：用户自评"有帮助" > 50%
  - 推荐：用户自发分享率
